# HL Design System — 데이터 표현 원칙

> 이 문서는 HL 내부 운영 도구의 **데이터 표현 구조**를 정의한다.
> 이 시스템의 핵심 차별점으로, 어떤 도구를 열어도 동일한 정보 흐름을 경험하게 만든다.
> AI가 데이터 화면을 만들 때 반드시 이 원칙을 따를 것.

---

## 1. 레이아웃 순서 원칙

모든 데이터 화면은 아래 순서를 따른다:

```
┌─────────────────────────────────────────────────────────────────┐
│  [1] Filter / Control Bar                                       │
│  필터, 검색, 날짜 범위, 상태 선택 등                              │
├─────────────────────────────────────────────────────────────────┤
│  [2] Summary / KPI                                              │
│  핵심 숫자 3~5개, 상태 요약, 변화 추이                             │
├─────────────────────────────────────────────────────────────────┤
│  [3] Main View (자유 시각화 영역)                                 │
│  차트, 그래프, 3D 뷰, 지도, 영상 그리드 등                         │
│  (도메인에 맞게 자유롭게 구성)                                      │
├─────────────────────────────────────────────────────────────────┤
│  [4] Detail / Table                                             │
│  정렬·검색·필터 가능한 테이블, 이벤트 리스트, 상세 정보              │
└─────────────────────────────────────────────────────────────────┘
```

### 규칙

- **[1], [2], [4]는 공통 구조**이다. 모든 내부 도구에서 일관되게 사용한다.
- **[3]은 자유 영역**이다. 도메인 특성에 맞게 시각화를 구성한다.
- 자유 시각화를 쓰더라도, **[4] 정형 상세 뷰(테이블)를 반드시 동반**한다.
- 데이터가 없는 경우 Empty State를 표시한다 ("데이터가 없습니다" + 안내 메시지).
- 단순 CRUD 화면이라도 이 순서를 가능한 한 따른다:
  - [1] 검색/필터 → [2] 건수 요약 → [4] 테이블 (자유 시각화 생략 가능)

> **이유**: 사내에서 10개의 서로 다른 도구를 사용해도, 정보를 찾는 흐름이 동일하면 사용자의 학습 비용이 제로가 된다.

## 2. Summary / KPI 규칙

### 구성 원칙

- 핵심 숫자는 **최대 5개 이하**. 그 이상은 "정보 과부하"이다.
- 각 KPI에 포함할 요소:
  | 요소 | 필수 | 설명 |
  |------|------|------|
  | 라벨 | 필수 | 무엇을 나타내는지 (예: "총 생산량") |
  | 값 | 필수 | 핵심 숫자 |
  | 단위 | 필수 | 건, %, 대, 개, 시간 등 |
  | 변화 방향 | 권장 | ↑ ↓ - 로 추이 표시 |
  | 상태 색상 | 권장 | 시맨틱 토큰으로 상태 표현 |

### 숫자 포맷 규칙

모든 내부 도구에서 숫자 표기를 통일한다:

| 유형 | 포맷 | 예시 |
|------|------|------|
| 정수 | 천 단위 구분자 | `1,234`, `56,789` |
| 소수 | 소수점 1~2자리 | `98.5%`, `3.14` |
| 퍼센트 | 소수점 1자리 | `92.3%` |
| 금액 | 천 단위 구분자 + 원 | `1,234,567원` |
| 시간 | `YYYY-MM-DD HH:mm:ss` | `2025-01-15 14:30:25` |
| 상대 시간 | 자연어 | `3분 전`, `2시간 전` |
| 큰 수 | 단위 축약 | `1.2만`, `3.4억` |

### KPI 카드 레이아웃

```
┌─────────────────┐
│ 총 생산량         │  ← 라벨 (text-secondary, text-sm)
│ 12,345 건        │  ← 값 + 단위 (text-2xl, font-bold)
│ ↑ 12.3%         │  ← 변화량 (success 또는 error 색상)
└─────────────────┘
```

- KPI 카드는 가로로 나열한다 (flex, gap-3).
- 카드 너비는 균등 분배 또는 콘텐츠 기반.
- 상태에 따라 좌측 border 또는 상단 border에 시맨틱 색상 적용 가능.

## 3. 시맨틱 색상 원칙

### 색상 = 상태 신호

내부 운영 도구에서 색상은 현장의 신호등과 같다. 의미 없는 색상 사용은 혼란을 초래한다.

| 색상 | 토큰 | 의미 | 운영 도구 사용 예 |
|------|------|------|------------------|
| 초록 | `success` | 정상, 가동, 완료 | 장비 가동 중, 작업 완료, 정상 수치 |
| 노랑 | `warning` | 경고, 주의, 점검 필요 | 임계치 근접, 유지보수 예정, 배터리 부족 |
| 빨강 | `error` | 에러, 정지, 위험 | 장비 고장, 시스템 다운, 이상 수치 |
| 파랑 | `primary` | 선택, 강조, 정보 | 현재 선택 항목, 활성 필터, 정보 안내 |
| 회색 | `gray` | 비활성, 대기, 해당 없음 | 미사용 장비, 비활성 항목 |

### 차트 색상 규칙

- 단일 시리즈: `primary-500` 사용.
- 다중 시리즈: **명도 차이**로 구분한다 (색조 변경 금지):
  ```
  시리즈 1: primary-500
  시리즈 2: gray-400
  시리즈 3: gray-600
  시리즈 4: primary-300
  시리즈 5: gray-300
  ```
- 상태를 나타내는 차트: 시맨틱 색상 사용 (정상=초록, 경고=노랑, 에러=빨강).
- 배경 그라데이션, 네온 차트, 화려한 색상 조합 금지.

> **이유**: 차트 색상이 도구마다 다르면 "이 초록은 뭘 의미하지?" 같은 혼란이 생긴다. 색상 의미를 고정하면 한 눈에 상태를 파악할 수 있다.

## 4. Density(밀도) 규칙

### 기본: Compact

내부 운영 도구는 정보를 한 화면에 최대한 담아야 한다. 장식보다 정보가 우선이다.

| 요소 | 값 | 비고 |
|------|-----|------|
| 테이블 row 높이 | 36~40px | `text-sm`(14px) 기준 |
| 카드 내부 패딩 | 12~16px | `space-3` ~ `space-4` |
| 카드 간 간격 | 12px | `space-3` |
| 섹션 간 간격 | 16~24px | `space-4` ~ `space-6` |
| 버튼 패딩 | 8px 12px (sm) / 8px 16px (md) | compact 기본 |

### 금지 사항

- Hero 섹션 금지 — 큰 배너, 이미지 영역은 정보를 밀어낸다.
- 장식 일러스트 금지 — SVG 아이콘은 허용, 대형 일러스트는 금지.
- 과도한 여백 금지 — `p-8`, `gap-8` 이상은 정보 밀도를 낭비한다.
- 큰 빈 공간 금지 — 화면의 50% 이상이 비어 있으면 안 된다.

> **이유**: 관제실 모니터에서 대시보드를 볼 때, 스크롤 없이 핵심 정보를 모두 확인할 수 있어야 한다.

## 5. Table First 원칙

### 테이블은 모든 데이터의 기본 뷰이다

- 어떤 데이터든 **테이블로 표현 가능해야** 한다.
- 차트, 3D 뷰, 지도 등 다른 시각화는 테이블의 **보조 수단**이다.
- 테이블 없이 카드만 나열하는 것은 금지 — 정렬/검색이 불가능하다.

### 테이블 필수 기능

| 기능 | 필수 여부 | 구현 방식 |
|------|----------|----------|
| 컬럼 정렬 | 필수 | 헤더 클릭 → asc/desc 토글 |
| 검색 | 필수 | Filter Bar의 검색 입력 연동 |
| 필터 | 필수 | 컬럼별 또는 전체 필터 |
| 페이지네이션 | 필수 (50건 이상) | 하단 페이지 네비게이션 |
| Row hover | 필수 | hover 시 배경색 변경 |
| 행 선택 | 권장 | 클릭 시 Detail 패널 연동 |

### 테이블 스타일 기준

```
헤더: bg-secondary, font-weight-semibold, text-xs, uppercase
셀:   text-sm, padding space-2 space-3
구분선: border-bottom (border 토큰)
hover: bg-tertiary
```

> **이유**: 차트는 추세를 보여주지만, 구체적 수치 확인과 데이터 탐색은 테이블이 압도적으로 효율적이다. 현장에서는 "정확한 숫자"가 필요하다.

## 6. 자유 시각화 + 정형 뷰 연결 규칙

### 자유 시각화는 허용하되 단독 사용은 금지

[3] Main View 영역에서 도메인에 맞는 시각화를 자유롭게 사용할 수 있다:
- 라인/바/파이 차트
- 3D 모델 뷰어
- 지도 (맵 뷰)
- 영상 그리드 (CCTV)
- 타임라인, 간트 차트
- 커스텀 SVG 다이어그램

### 연결 조건 (반드시 충족)

1. **선택 → 상세 연결**: 시각화 요소 클릭 시, [4] Detail 영역에 해당 항목의 상세 정보를 표시한다.
   ```
   예: 지도의 마커 클릭 → 우측 Detail Panel에 해당 장비 정보 표시
   예: 차트의 데이터 포인트 클릭 → 하단 테이블에서 해당 행 하이라이트
   ```

2. **테이블 fallback**: 시각화가 표시하는 데이터는 테이블로도 볼 수 있어야 한다.
   - 뷰 전환 토글 (차트 ↔ 테이블) 또는
   - 시각화 아래에 테이블이 항상 존재

3. **로딩/에러 상태**: 시각화 로딩 중에는 스켈레톤, 에러 시에는 에러 메시지를 표시한다.

> **이유**: 3D 뷰나 영상만 있고 상세 정보를 볼 수 없으면, 결국 다른 도구를 열어야 한다. 시각화와 정형 데이터가 연결되어야 원스톱 운영이 가능하다.

## 7. Filter / Control Bar 규칙

### 구성 원칙

- 화면 최상단에 위치한다.
- 가로로 나열하며, 공간이 부족하면 줄바꿈한다.
- 기본 구성:
  ```
  [검색 입력]  [상태 드롭다운]  [날짜 범위]  [적용 버튼]  [초기화]
  ```

### 필수 요소

| 요소 | 역할 | 필수 여부 |
|------|------|----------|
| 검색 입력 | 텍스트 기반 검색 | 필수 |
| 드롭다운/셀렉트 | 상태, 카테고리 필터 | 도메인에 따라 |
| 날짜 범위 선택 | 기간 필터 | 시계열 데이터 시 필수 |
| 적용 버튼 | 필터 실행 | 자동 적용이면 생략 가능 |
| 초기화 버튼 | 필터 리셋 | 필수 |

### 동작 규칙

- 필터 적용 시 [2] Summary와 [3] Main View, [4] Table 모두 연동 갱신.
- 활성 필터가 있으면 시각적으로 표시 (뱃지 카운트 또는 색상 변경).
- 필터 상태는 URL 파라미터에 반영하여 공유 가능하게 한다 (권장).

## 8. Empty State & Loading 규칙

### Empty State

데이터가 없을 때 빈 화면을 그대로 두지 않는다:

```
┌─────────────────────────────────┐
│         (아이콘)                 │
│    데이터가 없습니다              │
│    필터 조건을 변경하거나          │
│    데이터를 추가해 주세요          │
│                                 │
│    [필터 초기화]                  │
└─────────────────────────────────┘
```

### Loading State

- 테이블: row 자리에 스켈레톤 애니메이션.
- KPI 카드: 숫자 자리에 스켈레톤 바.
- 차트: 전체 영역에 스켈레톤 또는 스피너.
- 전체 로딩은 `duration-normal`(200ms) 이후에 스켈레톤 표시 (깜빡임 방지).

> **이유**: 빈 화면이나 무한 로딩은 "도구가 고장났나?"라는 혼란을 준다. 명확한 상태 표시가 운영 도구의 신뢰성을 높인다.
